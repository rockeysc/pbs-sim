Background

Prebid Server was built to solve the limits of browser-side header bidding.

What started as a fix for page latency and mobile limitations has grown into the foundation of modern programmatic infrastructure.

2014–2016
The starting point

Waterfall ends, header bidding begins

Publishers abandoned sequential waterfall auctions for simultaneous header bidding. Prebid.js launched in 2015 as the open-source standard—but browser costs quickly became a problem: too many bid requests, slower pages, and no clean path for mobile.

2017
PBS launches

Move the auction off the page

Instead of the browser calling 20 bidders, it calls one PBS endpoint—PBS fans out server-side. Two implementations emerged: PBS-Go (AppNexus, built for concurrency) and PBS-Java (Rubicon Project, built for enterprise scale).

2019–2021
Industry shift

Hybrid becomes the standard

Most publishers combined both: high-performing bidders stay client-side, latency-heavy or long-tail demand moves server-side. This balanced match rates, CPMs, page speed, and cost—and remains the dominant model today.

Key tech
Technical milestones

What PBS introduced

  • Stored requests — lightweight IDs replace large browser payloads
  • OpenRTB standardization across exchanges
  • Centralized bidder management and cloud-hosted infrastructure
Foundation

Prebid Server is an open-source solution for server-to-server header bidding.

Prebid.js orchestrates the auction in the browser, while Prebid Server handles server-side bid requests—enriching the request, enforcing policies, fanning out to bidders, and returning a normalized response.

Core idea Server-side bidding does not remove Prebid.js. It changes where bidder execution happens.
Input

Validate + enhance

PBS parses the OpenRTB request, resolves stored requests, and applies account and bidder-level defaults

Execution

Run server adapters

PBS calls multiple bidder adapters in parallel without loading them in the browser.

Output

Normalize results

PBS standardizes bid responses, applies currency conversion and caching, and returns them to Prebid.js for auction logic and targeting.

Prebid Universal Creative

The Prebid Universal Creative (PUC) is a JS snippet trafficked on every Prebid line item in the ad server. When a Prebid line item wins, the PUC reads Prebid.js targeting keys, retrieves the cached bid, and renders it—one creative that handles all bidders and all formats.

In plain English: The ad server only knows one trick — run this one little script. That script is the PUC, and its job is to look up whichever ad won the auction and actually put it on the page. It's the universal adapter between the ad server and every possible ad format.

Section 1

Prebid.js and Prebid Server introduce a separation of responsibilities between browser orchestration and server-side bidder execution

The architecture shifts from client-side bidder fan-out to a hybrid model, where Prebid.js continues to coordinate the page and auction flow, while Prebid Server executes selected bidders through an OpenRTB-based workflow.

Prebid.js

Client-side auction

  • Bidder adapters run in browser JavaScript.
  • Each bidder request originates from the page.
  • Browser cost grows as demand partners increase.
Prebid Server

Server-side auction

  • Server-side bidder execution
  • `s2sConfig` routes selected bidders to PBS.
  • The browser (via Prebid.js) sends a single request to PBS.
  • PBS performs bidder fan-out and returns normalized bid responses to Prebid.js
What stays the same

Regardless of where bidders run, Prebid.js still sets ad server targeting and owns the final auction decision.

Latency

PBS cuts browser network and CPU overhead by handling bidder calls server-side.

Protocol

PBS uses OpenRTB with ext.prebid for Prebid-specific metadata.

Operations

Client- and server-side demand can run together; Prebid.js merges bids and runs the auction.

Request Flow

Browser (Prebid.js) → Prebid Server → SSPs → Ad Server → Prebid Universal Creative → render.

This lifecycle highlights how Prebid.js orchestrates the page while PBS executes server-side bidders and returns normalized bids for the final auction and rendering in the browser.

Section 2

s2sConfig is the client-side configuration that routes selected bidders to Prebid Server.

It defines which bidders run in PBS, where requests are sent, and what Prebid-specific controls are included in the OpenRTB payload.

Illustrative `s2sConfig`
pbjs.setConfig({
  s2sConfig: [{
    accountId: 'freestar-account',
    bidders: ['appnexus', 'rubicon'], // server-side bidders
    enabled: true,
    endpoint: 'https://pbs.example.com/openrtb2/auction',
    syncEndpoint: 'https://pbs.example.com/cookie_sync',
    timeout: 650
  }]
});
How it fits in

Prebid.js reads `s2sConfig` at auction time and routes the listed bidders through a single OpenRTB request to the PBS endpoint instead of calling them directly from the browser.

Routing

`bidders`

Defines which bidders run server-side via Prebid Server.

Transport

`endpoint`

PBS OpenRTB auction endpoint (e.g. `/openrtb2/auction`).

Controls

Config options

Optional settings passed through s2sConfig to influence PBS behavior.

Section 3

Freestar uses a hybrid Prebid model where configuration is centralized and bidder execution can occur client-side or server-side.

Prebid Server acts as a server-side extension of Prebid.js, using the same OpenRTB-based auction framework while allowing selected demand partners to execute off-page.

Configuration

Centralized in FAD

  • SSPs are configured in FAD
  • Client or Server-Side selections are configured within FAD
Hybrid routing

Client-side or server-side

  • Some SSPs stay in Prebid.js.
  • Some SSPs are routed into PBS through `s2sConfig`.
  • Server-side bids are returned to Prebid.js for unified auction handling.
Final framingPrebid Server is not a separate stack—it is an extension of the Prebid ecosystem that shifts selected bidder execution to the server while keeping Prebid.js as the primary auction orchestrator.
Benefit

Reduced browser overhead

Fewer client-side requests and lighter front-end execution.

Benefit

Scalable demand

More bidders can participate without significantly increasing page weight.

Takeaway

Same auction, different execution paths

Freestar maintains the Prebid model while optimizing where bidder logic runs.

Bonus

Debug server-side bid requests directly in the browser's Network panel.

Appending ?fspb_debug=true to any Freestar-managed URL enables PBS debug mode, exposing every HTTP call PBS made during the auction under the request's preview tab.

Step 1

Enable debug mode

Append ?fspb_debug=true to the page URL and reload. This signals PBS to include a detailed debug object in the response.

Step 2

Filter network requests

Open DevTools → Network tab and filter by t13.io/openrtb2 to isolate the PBS auction request.

Step 3

Inspect HTTP calls

Select the request → Preview tab → expand the debug object. Under httpcalls, each bidder's outgoing request and response is listed.

Recap

Common misconceptions — and why Prebid Server still matters.

Server-side bidding isn't a replacement for client-side. Understanding the tradeoffs explains why hybrid has become the industry standard.

Misconception 1

"Server-side is always better"

Not necessarily. Moving bidders server-side comes with real tradeoffs:

  • Reduced cookie match rates
  • Lower user ID fidelity
  • Less browser signal access

That's why hybrid setups remain dominant.

Misconception 2

"Prebid Server replaced Prebid.js"

It didn't. Most publishers still use all of these together:

  • Prebid.js
  • Prebid Server
  • Ad server integrations
  • Identity frameworks
Why Prebid Server became important
  • Scaled header bidding beyond browser limits
  • Enabled mobile and app monetization
  • Reduced frontend complexity
  • Standardized server-side auctions via OpenRTB
  • Prepared publishers for privacy-driven web changes
