Prebid Server Concepts

What is Prebid Server?

Prebid Server is an open-source solution for server-to-server header bidding. It moves the auction logic from the browser to a server, reducing page latency and enabling more demand sources.

Key Terms

Header Bidding
An advanced programmatic technique where publishers offer inventory to multiple ad exchanges simultaneously before making calls to their ad servers.
SSP (Supply-Side Platform)
Technology platform that enables publishers to manage, sell, and optimize their available ad inventory.
OpenRTB
Open Real-Time Bidding protocol - an industry standard for programmatic advertising communication between buyers and sellers.
Price Granularity
Determines how bid prices are rounded into price buckets for targeting in the ad server.
Bid Adapter
Code that translates Prebid's internal auction format into a bidder's specific API format.
Floor Price
The minimum price a publisher will accept for an ad impression.
First-Price Auction
Winner pays exactly what they bid.
Second-Price Auction
Winner pays $0.01 more than the second-highest bid.

Workflow Phases

  1. Request Initiation: Publisher page constructs and sends ad request
  2. Bidder Adapters: Request is transformed for each enabled bidder
  3. Auction: Bids are collected and evaluated
  4. Post-Auction: Floors enforced, currency converted, winner selected
  5. Rendering: Winning creative is displayed

Bid Request

A bid request is an OpenRTB-formatted message sent by Prebid Server to each demand partner. It describes the ad opportunity in a standardised way so any compliant bidder can respond.

imp
Impression object — describes the ad slot (size, format, floor price) that the bidder is being asked to fill.
tmax
Maximum response time in milliseconds. Bidders that don't reply within this window are excluded from the auction entirely.
bidfloor
The minimum CPM the publisher will accept. Any bid below this value is rejected before the auction even begins.
ext.prebid
Prebid-specific extensions — carries targeting config, price granularity, enabled bidder list, and feature flags to the server.

Bid Response

A bid response is the OpenRTB reply a demand partner sends back to Prebid Server. It contains the price the buyer is willing to pay and the creative that will render if the bid wins.

seatbid
Bids grouped by buyer seat (DSP or SSP account). Each seat can submit one or more bids against the impression objects in the request.
price
The CPM the bidder is willing to pay. PBS compares this against the floor price and all other seat bids to determine the winner.
adm
Ad markup — the HTML, JavaScript, or VAST XML that gets rendered in the browser when this bid wins. Typically retrieved from Prebid Cache at render time.
ext.prebid.targeting
Key-value pairs (e.g. hb_pb, hb_bidder) sent to the ad server so it can identify and load the winning creative.
ext.prebid.cache
Cache entry URL where the creative is stored. The ad server calls this URL at impression time to retrieve the actual ad markup.

S2S Configuration

The s2sConfig object tells Prebid.js how to hand off auction work to Prebid Server instead of running adapters client-side in the browser.

adapter
"prebidServer" — the built-in Prebid.js module that packages the bid request and communicates with your PBS endpoint.
bidders
The list of demand partner IDs that PBS will call server-side. Bidders here bypass client-side JavaScript entirely, reducing page load time.
endpoint
The PBS auction URL. Accepts separate values for users who have given first-party consent (p1Consent) and those who have not (noP1Consent).
timeout / maxTimeout
timeout is the target response window; maxTimeout is the hard ceiling PBS enforces — any configured value above it is clamped.
syncEndpoint
The PBS cookie-sync URL. PBS calls this to align user identifiers across demand partners, improving bid match rates.
allowUnknownBidderCodes
When true, PBS passes through bids from adapters that aren't explicitly listed in bidders[] — useful for catching demand from alias adapters.
